# Telegram-бот «Кто я по знаку»

## Суть проекта

Это Telegram-бот для точного определения знака Зодиака.

Бот умеет:

* определять знак по обычной дате рождения;
* обрабатывать пограничные даты;
* уточнять время рождения;
* запрашивать место рождения;
* рассчитывать положение Солнца через Swiss Ephemeris;
* показывать профиль пользователя;
* вести лимит бесплатных проверок;
* сохранять лимиты в SQLite через Railway Volume.

Проект развивается постепенно. Главная цель — стабильный MVP, который можно безопасно расширять.

---

## Главное правило разработки

Мы не переписываем весь проект целиком.

Работаем только маленькими шагами:

1. Сначала проектируем.
2. Потом создаем или меняем один небольшой участок.
3. Если это новый модуль — сначала тестируем его отдельно.
4. Только потом подключаем к `bot.py`.
5. После каждого изменения запускаем бота.
6. После больших этапов проверяем:

   * обычный запуск;
   * Restart на Railway;
   * Redeploy на Railway.
7. После завершенного этапа записываем изменения в `CHANGELOG.md`.

Пользователь учится программированию.

Поэтому любые изменения нужно давать строго так:

1. Объяснить, зачем это делаем.
2. Показать конкретное место в файле.
3. Дать готовый блок кода.
4. Объяснить, почему именно туда.
5. Дать тест.
6. Только после теста переходить дальше.

Нельзя писать:

* «добавь где-нибудь»;
* «вставь примерно сюда»;
* «найди похожее место»;
* «перепиши весь файл», если это не маленький отдельный модуль.

Допустимо заменять весь файл только если файл небольшой и это снижает риск ошибок с отступами.

## Роли файлов

### `bot.py`

Главный файл Telegram-бота.

Отвечает за:

* команды `/start`, `/help`, `/clear`, `/feedback`, `/profile`;
* обработку сообщений;
* обработку inline-кнопок;
* сценарий ввода даты, времени и места;
* вызов функций расчета знака;
* отправку сообщений пользователю.

`bot.py` не должен знать, как именно хранятся данные.

---

### `user_profile.py`

Бизнес-логика профиля пользователя.

Отвечает за:

* получение профиля;
* проверку лимита;
* списание проверки;
* расчет остатка бесплатных проверок;
* текст профиля.

`user_profile.py` не должен писать SQL напрямую.

---

### `limits.py`

Настройки и тексты лимитов.

Содержит:

* `FREE_CHECKS_PER_MONTH`;
* `limit_text()`.

Если нужно изменить количество бесплатных проверок — менять здесь.

---

### `storage.py`

Слой хранения.

Он связывает бизнес-логику и базу данных.

Сейчас:

* `user_data` пока хранится в памяти;
* счетчики пользователей идут через SQLite.

Идея:
`user_profile.py` обращается к `storage.py`, а `storage.py` уже решает, откуда брать данные.

---

### `database.py`

Слой SQLite.

Отвечает за:

* подключение к базе;
* создание таблиц;
* чтение счетчиков;
* увеличение счетчиков;
* работу с Railway Volume.

Только этот файл должен знать SQL.

## SQLite и Railway Volume

SQLite работает через файл `bot.db`.

На Railway обычная файловая система контейнера не подходит для постоянных данных:

* при Restart данные сохраняются;
* при Redeploy без Volume данные сбрасываются.

Поэтому база должна лежать на Railway Volume.

В `database.py` путь выбирается так:

1. Если есть `RAILWAY_VOLUME_MOUNT_PATH`, база пишется туда.
2. Если есть `DATABASE_PATH`, используется он.
3. Если ничего нет, локально используется `bot.db`.

При запуске бота обязательно вызывается:

```python
init_db()
```

Это нужно, чтобы таблицы создавались автоматически на новом деплое.

### 1. `bot.py` не должен разрастаться

Не добавлять туда лишнюю бизнес-логику.

Если логика касается:

* профиля — в `user_profile.py`;
* лимитов — в `limits.py`;
* хранения — в `storage.py`;
* SQL — в `database.py`.

Новые модули добавляются только если они реально упрощают проект.

### 3. Сначала стабильность, потом новые функции

2. SQLite.

## Методика проверки после крупных этапов

После каждого большого этапа обязательно проверять:

### 1. Обычная работа

* `/start`
* `/profile`
* обычная дата
* пограничная дата с временем
* пограничная дата без времени

### 2. Restart Railway

После Restart:

* бот должен запуститься;
* счетчик проверок должен сохраниться.

### 3. Redeploy Railway

После Redeploy:

* бот должен запуститься;
* счетчик проверок должен сохраниться;
* база не должна пересоздаваться с нуля.

## Что нельзя делать

Нельзя:

* переписывать весь `bot.py`;
* возвращать Premium-разделы;
* добавлять оплату раньше времени;
* смешивать SQL и Telegram-логику;
* удалять рабочий код без причины;
* делать много изменений без теста;
* переходить к следующему шагу, если предыдущий не проверен.

---

## Что делать при ошибке

Если появляется ошибка:

1. Не чинить наугад.
2. Найти конкретную строку из traceback.
3. Попросить или посмотреть кусок кода вокруг ошибки.
4. Исправить минимальный участок.
5. Запустить бота.
6. Только потом продолжать.
